# Question Bank — 案件キックオフ + 事前資料/素材 + シート方式 30 問 + 事業語言い換え

中小企業の経営者・マーケター向け。マーケ用語を避け、事業語で聞く。

ヒアリング全体の流れ:

```
⓪ キックオフ 3 問（対話）
  └─ orchestrator に case doc 作成依頼（非同期）
① 事前共有資料リクエスト（一括メッセージ・URL リスト中心）
② 制作素材リクエスト（一括メッセージ・URL or アタッチ）
③ コアシート 10 問（1 メッセージで一括送信）
  └─ 回答受信 → hearing doc 更新
④ 詳細シート 20 問（1 メッセージで一括送信）
  └─ 回答受信 → hearing doc 更新
⑤ coverage-check → hearing-report → orchestrator にハンドオフ
```

---

## ⓪ 案件キックオフ確認（最初に対話で 3 問）

ヒアリング本体に入る前に対話形式で 3 つだけ押さえる。**納期は B/H ブロックで会話の流れで確認するため、ここでは聞かない**。

1. 「まず軽く 3 つだけ。クライアント企業名（または案件名）を教えていただけますか？」
2. 「サイトの種類は LP（1枚もの・キャンペーン用）と、サービス/SaaS の紹介サイトのどちらに近いですか？」
3. 「この案件のオーナー（最終判断する人）はどなたですか？」（自分なら「自分」と回答）

これら 3 つが揃ったら **裏で `site-strategy-orchestrator` に session run** で case doc 作成を依頼（非同期）し、クライアントには「ありがとうございます。それでは事前資料の確認に進みますね」と続ける。

orchestrator へのハンドオフ依頼テンプレ:
```
aachat session run site-strategy-orchestrator --project <project> --via claude-code "
新規案件の case doc を作成してください。
- クライアント名: <name>
- case_id 候補: <kebab-case>
- brief_type: <lp | saas>
- オーナー: <human-username>
- 現フェーズ: hearing（私が継続中）

期待: cases/<case-id>.md を作成。
- assignee は site-hearing-agent のまま
- status は hearing
- due_date は未確定（後で hearing-agent から session send で追記する）
- pre_shared_assets: []
- creative_assets: []
"
```

## ① 事前共有資料リクエスト（一括メッセージ）

ヒアリングだけでは拾えない情報（商品スペック・実顧客の声・既存運用データ等）を **URL リスト中心** で集める。詳しくは `knowledge/pre-shared-assets-checklist.md` 参照。

クライアントに送る一括メッセージのテンプレ（シートテンプレ全文は `knowledge/sheet-templates.md`）:

```
ヒアリングを始める前に、もし手元にある資料があれば URL を共有いただけると、後の戦略設計とリサーチの精度が大きく上がります。
（全部揃わなくて OK です。「ある分だけ」共有してください）

▼ 事業を理解するための資料
1. 既存サイト URL（自社の現サイト・LP・サービスページ）:
2. 商品/サービスの紹介ページ URL（カタログや料金表でも可）:
3. 過去の提案資料・営業資料の URL（あれば）:
4. お客様の声・レビュー・事例の URL（自社サイト掲載 or 外部レビュー）:

▼ 競合・参考を理解するための資料
5. 直接の競合だと思うサイト URL（2〜3 つ）:
6. 「このサイト好き」と思う参考サイト URL（2〜3 つ）:
7. 「これにはしたくない」サイト URL（あれば）:

▼ 追加情報（任意）
8. 業界レポート・調査資料の URL（あれば）:
9. その他、見ておいてほしい情報の URL:
```

クライアントが回答したら、case doc の `pre_shared_assets` に **session send** で逐次追記する（orchestrator に依頼）。

## ② 制作素材リクエスト（一括メッセージ）

サイト制作チームが後で困らないよう、**最初に**素材の所在を確認しておく。詳しくは `knowledge/creative-assets-checklist.md` 参照。

クライアントに送る一括メッセージのテンプレ:

```
続いて、サイトに使う素材についても今のうちに確認させてください。
（揃ってないものは「これから作る」と書いていただいて OK です）

▼ ブランド系
1. ロゴデータ（AI/SVG/PNG）の URL or アップロード:
2. ブランドガイド（フォント・色・トーン）の URL or アップロード:
3. ブランドカラー（メイン/サブ。HEX で）:
4. 使用フォント指定（あれば）:

▼ ビジュアル素材
5. 商品写真・サービス画像の URL or 共有方法:
6. 動画素材（紹介動画・お客様の声等）の URL:
7. 撮影予定の素材（必要なら何を撮るか）:

▼ 必須要素
8. 必ず載せたい要素（パートナーロゴ / 認証バッジ / 受賞歴 等）:
9. 法令・規制で必須の表記（特商法・薬機法対応など）:
```

クライアントが回答したら、case doc の `creative_assets` に **session send** で逐次追記する。

## ③ コアシート 10 問（1 メッセージで一括送信）

戦略ブリーフの core fields に直結する最重要 10 問。クライアント自身のペースで埋めて返してもらう。

クライアントに送るシート（全文は `knowledge/sheet-templates.md`）:

```
それでは本ヒアリングです。まず最重要の 10 問をまとめて送ります。
順番通りでなくて OK、書ける項目から書いてください。1〜2 文ずつで十分です。

【コアシート（10 問）】

事業の現在地
Q1. このサービスを一言で言うと？（社内の人に説明するように）

JTBD（決定打）
Q2. 最近契約してくれたお客さんは、契約前は何で困っていましたか？
Q3. そのお客さんが「これだ」と決めた瞬間、何が決め手でしたか？
Q4. もしウチがなかったら、お客さんは何で代用しますか？（具体名で）

ペイン / ゲイン
Q5. お客さんがよく口にする愚痴・困りごとは？（生の言葉のまま）
Q6. 契約後に「これは助かった」と言われたことは？

サイトの役割
Q7. このサイトに来た人に、最終的にしてほしい 1 つの行動は？
Q8. 月にどれくらい問い合わせ / CV があれば「成功」と言えますか？

オファー
Q9. 価格はどう設定していますか？（プラン構造・レンジ）

ブランド
Q10. ブランドの「らしさ」を一言で言うと？
```

回答受信後:
- 発話ログにそのまま `[<client-name>] "..."` で記録
- 構造化要約セクションを更新
- coverage の主要 6 セル状況を評価
- 抽象的すぎる回答があれば、詳細シートに ⚠️ マークで深掘りメモを追加

## マーケ用語 → 事業語 言い換え辞書

| マーケ用語 | 事業語の言い換え |
|---|---|
| ターゲットペルソナ | 「一番来てほしいお客さん」を一人、名前を付けて教えてください |
| バリュープロポジション | お客さんが「これウチに必要」と感じる一番のポイントは？ |
| KPI | 「今月よかったね」と言える数字は何ですか？ |
| ペインポイント | お客さんから一番よく聞く愚痴は？ |
| 競合優位性 | 他社さんと比べて「ここは負けない」と思うのは？ |
| CTA | サイトを見た人にまず何をしてほしいですか？ |
| エンゲージメント | サイトを見てる時にお客さんが「これ気になる」と思うやつを増やしたい、で合ってますか？ |
| アウェアネス | 「ウチをまだ知らないお客さん」と「もう知ってるお客さん」、どっちが多いですか？ |
| インサイト | お客さんが「言葉にしてないけど、たぶんこう思ってる」と感じることは？ |
| ICP（Ideal Customer Profile） | 一番ハマる「典型的な良いお客さん」の業種・規模・状況を 1 つだけ |
| ファネル | お客さんが「知る → 検討する → 買う」の 3 段階で、どこが一番詰まりますか？ |

## 質問運用ルール

- **キックオフ 3 問（⓪）は対話、ヒアリング 30 問（③④）はシート方式**
- シートを送る時は **1 メッセージで一括**。1 問ずつ畳み掛けない
- 回答が抽象的・短すぎる場合のみフォロー質問する（畳み掛けない）
- マーケ語を口にしたら一拍置いて事業語に言い換える
- 個人情報・機密が混入したら「これは載せても大丈夫ですか？」と確認
- 既存サイト URL や顧客レビュー URL を受け取ったら、要件整理段階で research-agent が deep_scan するので、必ず case doc に追記する
